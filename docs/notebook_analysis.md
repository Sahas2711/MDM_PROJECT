### Reproducibility Assessment
**Confidence: Medium**

Positive indicators:
- Export code is structured and complete
- All artifacts present in backend/models/ (except VAE weights and GB model)
- Feature columns and label encoders exported — input pipeline is reproducible

Risks:
- Notebook uses Google Colab (`from google.colab import files`) — not runnable locally without modification
- Hardcoded paths like `/content/exported_models` — Colab-specific
- `optimal_k` variable referenced in KMeans export — must be defined earlier in notebook; if cells run out of order, this fails
- VAE architecture must be copy-pasted from training code to inference code — not self-contained
- DNN accuracy of 63.75% vs static UI claim of 99.1% — suggests either different evaluation split or the UI number is fabricated

### Sequential Execution Risk
**Confidence: Medium**

- Export cell references `classification_models['Random Forest']`, `scaler`, `feature_cols`, `label_encoders`, `ann_model`, `dnn_model`, `vae`, `optimal_k`, `X_scaled`, `KMeans` — all must be defined in prior cells
- If any training cell is skipped or fails, the export cell will raise NameError
- VAE export uses `vae.save_weights()` not `vae.save()` — requires architecture reconstruction at inference time (not done in backend)

### Hidden State Issues
- `optimal_k` is used in KMeans export but not defined in the export cell — depends on prior cell execution
- `X_scaled` used in KMeans fit — depends on preprocessing cells
- `classification_models` dict — depends on training loop cells

### Broken Pipeline Risks
- **VAE weights exported but never loaded in backend** — complete dead end
- **DNN accuracy discrepancy** — 63.75% in notebook vs 99.1% in UI static chart — one of these is wrong
- **fruit_classifier.h5 not in notebook export list** — this model exists in backend/models/ but is not exported by the notebook's export cell; its training source is unknown
